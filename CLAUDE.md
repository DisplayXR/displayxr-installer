# CLAUDE.md

Project guidance for Claude Code (claude.ai/code) when working in this repo.

## What this repo is

`displayxr-installer` produces the **end-user meta-installer** for DisplayXR — one `.exe` / `.pkg` per OS that chains every component installer (runtime + Shell + Leia plug-in + MCP Tools) into a single guided flow.

Tracks issue [`DisplayXR/displayxr-runtime#284`](https://github.com/DisplayXR/displayxr-runtime/issues/284).

**Not in this repo:**
- The per-component installers — each component publishes its own installer to its own GitHub Release. This repo only assembles them.
- The dev-orchestrator scripts (`scripts/setup-displayxr.{sh,bat}`) — those live in `DisplayXR/displayxr-runtime`. The bundle is the *user-facing* counterpart to the dev orchestrator.

## Issue tracking

All issues for this work live on `DisplayXR/displayxr-runtime` (issue numbers like `#284`). This repo does not maintain its own issue tracker. Commit subjects: `<type>(#NN): <subject>`.

## Cross-repo dependencies

| Source repo | What we consume | How |
|---|---|---|
| `DisplayXR/displayxr-runtime` | `scripts/lib/components.sh` (per-component asset table) | `curl https://raw.githubusercontent.com/.../$RUNTIME_TAG/scripts/lib/components.sh` |
| `DisplayXR/displayxr-runtime` | `DisplayXR-Installer-*.pkg` (macOS), `DisplayXRSetup-*.exe` (Windows) | `gh release download $RUNTIME_TAG --repo ...` |
| `DisplayXR/displayxr-shell-releases` | `DisplayXRShellSetup-*.exe` (Windows) | `gh release download $SHELL_TAG --repo ...` |
| `DisplayXR/displayxr-leia-plugin` | `DisplayXRLeiaSRSetup-*.exe` (Windows) | `gh release download $LEIA_TAG --repo ...` |
| `DisplayXR/displayxr-mcp` | `DisplayXRMCPSetup-*.exe` (Windows) | `gh release download $MCP_TAG --repo ...` |

Components without a macOS `.pkg` today (Shell, Leia, MCP) are silently skipped on the macOS side. The bundle build picks them up automatically when their releases start carrying `.pkg` artifacts and the runtime repo's `components.sh` populates `COMPONENT_PKG_MACOS_<name>`.

## Build commands

```bash
# macOS
./scripts/build-bundle.sh --version v0.1.0
# → _out/DisplayXRBundle-0.1.0.pkg

# Windows (from non-elevated cmd; the bundle itself elevates at install time)
scripts\build-bundle.bat --version v0.1.0
:: → _out\DisplayXRBundle-0.1.0.exe
```

Prerequisites:
- **macOS:** `gh` (with auth), `jq`, `productbuild`, `pkgutil` (Xcode CLT)
- **Windows:** `gh` (with auth), NSIS 3.x (`choco install nsis`), PowerShell

## Release flow

`workflow_dispatch` on `.github/workflows/publish-bundle.yml` — release engineer triggers manually with a tag input (e.g. `v0.1.0`). Workflow:
1. Builds macOS `.pkg` and Windows `.exe` in parallel
2. Creates GitHub Release with the input tag
3. Attaches both artifacts via `softprops/action-gh-release@v2`

**Not** auto-fired on component releases (would re-publish untested bundles).

The bundle's `versions.json` is the source of truth for which component versions ship together in a given bundle tag. Bumping a component pin in `versions.json` is the precondition for a new bundle release.

## Conventions inherited from `displayxr-runtime`

- **Commit messages:** `<type>(#NN): <subject>`. Types: `feat`, `fix`, `chore`, `docs`, `refactor`. Always reference the runtime-repo issue number.
- **PR title under 70 chars.** Detail goes in the body.
- **Co-Authored-By trailer for Claude commits.**
- **No -i flags on git** (no interactive rebase/add).
- **License:** Boost Software License 1.0 — same as runtime + shell + plugins.

## Things to be careful about

- **Don't strip ad-hoc signatures** from macOS dylibs when re-wrapping the runtime `.pkg`. `pkgutil --expand` / `pkgutil --flatten` preserve signatures by default; do NOT add `install_name_tool` or `codesign --remove-signature` to the build flow. Regression of `DisplayXR/displayxr-runtime#279` would SIGKILL the installed runtime at `dlopen`. The build script verifies the runtime payload's presence after re-wrap; extend that check if you change the rewrap flow.
- **Don't auto-fire `publish-bundle.yml`.** It's `workflow_dispatch` for a reason (compat-test gate). The drift-check workflow exists to *warn* on schema drift, not to bump pins.
- **Don't sign anything.** v1 ships unsigned (#280 / #281 in runtime repo are parallel work). When those land, signing gets a small CI block; until then, no `codesign` / `signtool` calls.
